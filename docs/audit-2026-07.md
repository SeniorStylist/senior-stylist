# Full Site Audit — 2026-07-06

Four-dimension sweep (security / correctness / efficiency / bloat) by parallel review
agents, findings adversarially verified against CLAUDE.md's load-bearing invariants,
then fixed in priority order. Four themed commits, `tsc --noEmit` clean after each.

**DB migration in this pass:** `drizzle/0021_qb_payments_pi_unique.sql` (unique partial
index on `qb_payments.stripe_payment_intent_id`). Self-bootstrapped by
`src/lib/payments-ddl.ts` on any payments-route call, so the deploy is safe either way —
but apply it in Supabase for cleanliness:
`CREATE UNIQUE INDEX IF NOT EXISTS qb_payments_stripe_pi_unique ON qb_payments (stripe_payment_intent_id) WHERE stripe_payment_intent_id IS NOT NULL;`

## Fixed — Security (`fix(security)` commit)

| # | Severity | Finding | Fix |
|---|---|---|---|
| S1 | Medium | **OCR import IDOR** — `POST /api/log/ocr/import` accepted `residentId` / `serviceId` / `additionalServiceIds` UUIDs without facility-scope checks (only `stylistId` was guarded). A forged id could attach bookings to — and mutate the room number of — another facility's resident. | Provided ids now validated against the target facility's loaded resident/service lists (mirrors the stylist guard); 403 on mismatch. Include-only entries checked. |
| S2 | Low | `charge.ts` — explicit `paymentMethodId` lookup wasn't scoped to the resident; a caller could point a charge at a card vaulted for a different resident. | Lookup now requires `paymentMethods.residentId = resident.id`. |
| S3 | Low | `saveCardFromSetupIntent` never asserted the SetupIntent belonged to the resident — a foreign `setupIntentId` could vault someone else's card under the wrong resident. | The SI's Stripe customer must equal the resident's stored `stripe_customer_id`. Covers both the client POST and webhook paths. |
| S4 | Low | Portal `[token]/checkout` — `bookingId` went into Stripe metadata unvalidated; the webhook marks that id paid, so a token holder could flip ANY booking (any facility) to paid. | Booking must belong to the token's resident + facility before the Checkout session is created. |
| S5 | Info | `residents/[id]/photo` POST returned the raw storage path, violating the documented "signed URLs only" invariant. | Response now returns only the signed URL (no client used the path). |

Everything else in the security sweep verified clean: role guards + facility scoping on
mutation routes, Zod `safeParse` coverage, rate-limit coverage, webhook signature
verification, RLS invariants, `NEXT_PUBLIC_` exposure, cron Bearer checks.

## Deferred (known, intentional — with reasons)

- **Recharts static imports ×3** (`analytics/reports-client`, `reports/reports-client`,
  `master-admin/reports-tab`): the documented pattern requires a single `dynamic()`
  wrapper component per chart (per-export dynamic was tried and reverted — commit
  6d2c300). Real refactor across three analytics pages; impact is limited to those
  routes' own bundles. Do as its own pass.
- **Master reports-tab date formatting** uses browser tz (bookings span facilities; the
  API would need to ship per-facility tz). Cosmetic ±1 day at month boundaries for a
  master-only view.
- **Week/month window math in server-local (UTC)** on `my-account` + `stylists/[id]`
  pages — aggregation edge case for evening bookings near week boundaries; needs a
  facility-tz week-anchor helper.
- **Tier-3 efficiency**: `recomputeFacilityBalances` per-facility loop (typically 1
  facility per call), `super-admin/reports/monthly` per-facility Promise.all (cached
  5min, master-only), auto-match ≤500-credit transaction loop (master-only manual op).
  All low-traffic; transactional clarity favored.
- **`merge-facilities` per-table loops** — master-only one-off tool; correctness of the
  single transaction matters more than speed.
- **Inline fuzzy copies** in `log/ocr/import` + `residents/duplicates` routes: NOT
  identical to canonical `fuzzy.ts` (no STOP_WORDS/`#`-stripping/substring boosts).
  Consolidating CHANGES matching behavior — needs deliberate threshold re-testing, not
  an audit drive-by.
- **Candidate-dead routes kept**: `GET /api/facilities/admin-contact`,
  `POST /api/bookings/sync` — no web-client caller found, but both are documented and
  may serve ops tooling; confirm with Josh before removal.
- **Over-exported live functions + type-only exports** (cosmetic `export` keywords) —
  inventory in the bloat agent report; zero runtime impact.
- **Oversized clients** (inventory only): `log-client.tsx` (1875 lines),
  `ocr-import-modal.tsx` (1558 lines) — refactor candidates when next touched.
